# Plan 2 v0.2.1 Audit Remediation Design

## Scope

This design fixes every finding from the post-release Plan 2 audit that can be
resolved inside the Tool Calling + Simple Agent Loop boundary. The work is a
patch for the already published `v0.2.0` release and prepares `v0.2.1`; the
existing `v0.2.0` tag must not be moved, deleted, or recreated.

The patch covers three cohesive remediation areas:

1. Tool input, filesystem, permission, and resource boundaries;
2. Simple Agent execution limits, timeout behavior, and ToolCall ordering;
3. Agent workspace recovery, focused UI tests, Registry setup, and release
   truth corrections.

The patch does not add RAG, Embedding, Memory, MCP, Shell/file-writing Tools,
Human Approval, Agent Runtime v2, AgentStep/Trace persistence, run listing,
polling, cancellation, resume, retry, or a real Provider/network acceptance
test. `web_fetch` remains deferred and has no runtime surface or dependency.

The user owns staging, committing, pushing, and tag creation. Codex must not
perform any Git mutation beyond editing the requested working-tree files.

## Verified Root Causes

| Finding | Root cause | Required correction |
|---|---|---|
| Private-key filtering misses `.ppk`, `.p8`, `id_ed25519_sk`, and arbitrary names | Security policy relies only on a small component-name denylist | Expand conservative filename rules and inspect UTF-8 file bytes for recognized private-key headers before decoding or returning content |
| `max_steps` allows one Provider response to execute an unbounded ToolCall batch | The loop limits Provider decisions while `_execute_tool_round` iterates every returned call | Define `max_steps` as the maximum number of Tool executions, reject an over-budget round atomically, and allow one final Provider response after the budget is consumed |
| Tool arguments can be arbitrarily large and are persisted before validation | Built-in path schemas have no `maxLength`; `_record_tool_call` copies raw Provider arguments into SQLite first | Add a 4096-character path bound, enforce a global 64 KiB UTF-8 JSON argument bound, and persist only validated arguments; invalid/unknown/blocked calls store `{}` |
| Tool public contracts accept NaN/Infinity | Pydantic accepts Python floats unless standard-JSON finiteness is checked explicitly | Use one shared standard-JSON validator for ToolCall arguments and ToolResult data/metadata |
| `list_dir` materializes an entire directory | `sorted(scandir(...))` runs before the returned-entry cap | Scan at most `max_entries + 1` children per visited directory, sort only that bounded window, and report truncation |
| Filesystem docs promise no symlink traversal while `read_file` follows internal links | Resolution occurs before link components are rejected | Reject every user-supplied symlink or Windows reparse-point component for both built-in file Tools; continue reporting but never traversing links found during directory listing |
| `permission_level` is informational | Agent dispatch never checks it | Allow execution only for `read_only`; record every other registered Tool as `blocked` with a safe result and no invocation |
| Agent execution has no total deadline | Only Provider-adapter and individual Tool timeouts exist | Add a configurable whole-run deadline with a safe failed AgentRun result |
| ToolCall order depends on timestamps and UUIDs | No strict persisted sequence exists | Add a one-based `sequence_index`, backfill existing rows per AgentRun, enforce uniqueness, and expose it through API/frontend |
| The default setup has no safe Tool-capable Registry workflow | Only tracked `models.json` is loadable and intentionally declares no Tool support | Add an optional local Registry path setting plus a tracked secret-free example copied to an ignored local file; never mark the generic tracked model Tool-capable |
| A completed request can become inaccessible after leaving Agent | The request gate correctly prevents stale UI/URL writes but discards the returned run ID | Save only the returned run ID in tab-scoped session storage before the stale-view check and restore it when Agent is reopened without a URL run ID |
| Tool arguments are rendered in full and a ToolCall ID is mislabeled | Result summaries are bounded but arguments are not; UI copy calls the correlation ID a Provider call ID | Apply the same 600-character summary boundary to arguments and label the field `Tool Call ID` |
| AgentPage lacks mounted async tests | Tests cover helpers and server-rendered child components only | Add jsdom-based React DOM tests for model loading, submission, URL restore, leave/reopen recovery, and transport/structured failure paths |
| Release documentation still describes pre-release state | The release handoff files were not truth-corrected after the user published the commit and annotated tag | State that `v0.2.0` is published, prepare `v0.2.1`, close S8/tag-pending text, and retain only real limitations |

## Tool Security Design

### Standard JSON And Argument Limits

A focused `app.tools.json_values` module will own a helper that round-trips
validation through `json.dumps(..., allow_nan=False)`. It accepts only values
that can be represented by standard JSON and optionally enforces a maximum
encoded UTF-8 byte length. It never includes the rejected value in its error.

`validate_tool_arguments` will call this helper before JSON Schema validation
with `MAX_TOOL_ARGUMENT_JSON_BYTES = 65_536`. `ToolCallRequest.arguments`,
`ToolResult.data`, and `ToolResult.metadata` will use the same rule. This keeps
the public Tool contracts, Agent boundary, and persistence boundary aligned.

Both built-in Tools will declare `path.maxLength = 4096`. This is a schema
contract visible to Providers and produces the existing safe invalid-argument
result when exceeded.

### Sensitive Files And Links

Filename filtering will cover common private-key containers and SSH security
key names, including `.ppk`, `.p8`, `.p12`, `.pfx`, `.jks`, `.keystore`, and
`id_<algorithm>_sk` variants. Before UTF-8 decoding, `read_file` will reject
content containing recognized PEM/OpenSSH/SSH2/PGP private-key headers or a
PuTTY private-key file header. Tests use only synthetic markers and temporary
files.

`resolve_workspace_path` will inspect each lexical component below the resolved
workspace root using non-following metadata. Any symbolic link or reparse point
in the user-supplied path is rejected before the final target is used. Missing
paths continue to reach the existing not-found handling. Directory listings may
show a child as `symlink`, but never recurse into it.

`list_dir` will consume at most `max_entries + 1` directory entries at each
visited level, sort that bounded sample, and stop globally as soon as the
result limit is exceeded. A truncated listing remains explicitly marked. Small
directories preserve the current deterministic alphabetical behavior.

## Simple Agent Design

### Execution Budget

`SimpleAgentRequest.max_steps` remains the public field for compatibility but
is documented and implemented as the maximum number of Tool executions. A
direct answer needs one Provider call. A run that executes up to N ToolCalls may
make one final Provider call, so the maximum Provider decisions are N + 1.

Before a Tool round begins, the service compares the entire response batch with
the remaining Tool budget. If the batch does not fit, the run fails with the
existing maximum-steps error and executes/persists none of that batch. Partial
execution is forbidden because a Provider protocol turn requires observations
for every ToolCall it emitted.

### Validation, Permission, And Persistence

For each accepted ToolCall, the service resolves the Tool, checks
`permission_level == "read_only"`, and validates arguments before constructing
the database row:

- valid read-only Tool: persist the validated defensive copy and execute it;
- unknown Tool: persist `{}`, status `failed`, and a safe unavailable result;
- invalid or oversized arguments: persist `{}`, status `failed`, and a safe
  validation result;
- non-read-only Tool: persist `{}`, status `blocked`, and a safe permission
  result without invoking the Tool.

This is a Plan 2 deny-by-default dispatch boundary, not Human Approval.

Each accepted row receives the next one-based `sequence_index` in execution
order. A migration backfills old rows per AgentRun using `created_at` and `id`,
then adds a non-null constraint and a unique constraint on
`(agent_run_id, sequence_index)`. Migration verification uses only a newly
created temporary SQLite database.

### Whole-Run Timeout

`AGENT_RUN_TIMEOUT_SECONDS` will default to 120 seconds and be constrained to a
finite positive operational range. The Simple Agent loop runs inside this
deadline. Expiry produces a terminal failed AgentRun with `Agent run timed out`,
preserving ToolCalls completed before expiry and never exposing the underlying
exception. External cancellation remains cancellable and is not converted into
a normal failure.

Individual Tool timeouts remain in force. Their `asyncio.to_thread` limitation
is still cooperative for blocking filesystem work and remains documented.

## Registry And Frontend Design

### Local Registry Override

`MODEL_REGISTRY_PATH` will optionally select a JSON Registry file. An empty
setting continues to load the packaged `models.json`. A tracked
`models.local.example.json` contains only synthetic Provider/model names and
shows `supports_tools=true`; README instructions copy it to the ignored
`models.local.json`, replace the model identifier with the user's configured
Provider model, and point `.env` at it. No credential is stored in Registry
JSON, and the generic tracked default stays `supports_tools=false`.

### Recoverable Synchronous Runs

The frontend stores only the latest returned AgentRun UUID in `sessionStorage`.
The storage write happens when POST resolves, even if the Agent component has
already unmounted. URL and component state remain protected by the existing
request gate. Reopening Agent without `?run=` restores the stored UUID through
the existing GET endpoints; an explicit URL run ID has priority. Choosing
`New task` clears both URL and stored ID.

This does not create a run list, polling, cancellation, background execution,
or cross-tab state.

### Rendering And Tests

ToolCall arguments use the existing deterministic JSON formatter followed by
the 600-character summary helper. The correlation field is labeled
`Tool Call ID`; `sequence_index` is displayed separately. No Provider request
ID is invented.

Mounted AgentPage tests use React DOM with a per-file jsdom environment and
module-level API mocks. They verify:

- Tool-capable model loading and form enablement;
- successful submit, result rendering, URL replacement, and session storage;
- initial URL restore and ToolCall fetch;
- unmount during POST, no stale URL mutation, and later session recovery;
- model-load failure, structured Agent failure, and transport error states.

Only `jsdom` is added as a frontend development dependency; production bundles
do not include it.

## Documentation And Version Design

Backend OpenAPI metadata, backend package metadata, frontend package metadata,
lockfile, release-version tests, and public docs will be synchronized to
`0.2.1`. CHANGELOG will describe the security and correctness patch. Until the
user creates the tag, documentation will distinguish the published `v0.2.0`
release from the prepared `v0.2.1` working-tree candidate.

README, README_CN, project overview, architecture, Tool design, Agent loop/API,
Plan 2 release summary, execution table, and final review/handoff records will
be corrected where they still claim `v0.1.0` is latest or `v0.2.0` tagging is
pending. Existing release screenshots will be regenerated only through local
mock data if visible labels or sequence metadata change.

## Test-First Acceptance Matrix

| Requirement | First failing evidence | Completion evidence |
|---|---|---|
| Private-key and link denial | Synthetic `.ppk`, `id_ed25519_sk`, arbitrary-name PEM, and internal symlink reads currently succeed | All return the existing safe path/content denial without leaking markers |
| JSON/path/argument bounds | NaN and a 1,000,000-character path currently validate | Public schemas and Agent validation reject them; invalid raw values are not persisted |
| Tool execution cap | One Provider response currently executes 25 calls with `max_steps=2` | Over-budget round executes zero calls; an exactly-budgeted round completes after a final Provider response |
| Permission boundary | A registered non-read-only Tool currently executes | It is never invoked and produces a `blocked` ToolCall |
| Total timeout | A never-returning mock Provider has no Agent-level deadline | Run ends safely at a small injected test deadline |
| Bounded directory scan | `list_dir` consumes a whole mocked iterator | It requests no more than `max_entries + 1` children per level |
| Strict sequence | ToolCalls have no persisted order field | Fresh migration, ORM, service, API, and frontend expose one-based stable order |
| Local Registry workflow | Dependency always loads packaged Registry | Temporary local Registry path is selected without editing tracked defaults |
| UI argument bound/label | Full arguments and `Provider call ID` are rendered | Arguments are summarized and labels/sequence are accurate |
| Async page behavior | No mounted test exercises effects/submission | jsdom tests cover load, submit, restore, leave/reopen, and errors |
| Release truth | Docs claim the previous release/tag state | Version surfaces and release narrative agree with Git history and patch status |

## Verification And Review

Focused red/green cycles must precede each production change. Final verification
must include:

- full backend pytest and `pip check`;
- fresh temporary SQLite Alembic upgrade, head check, and migration consistency;
- frontend full Vitest, typecheck, and production build;
- local mock Agent smoke and updated desktop/mobile screenshots if required;
- Markdown local-link scan, secret scan, generated-artifact scan, `web_fetch`
  deferral scan, later-Plan boundary scan, Git status/staged paths, and
  `git diff --check`.

Codex self-review is the only review gate. Findings are classified as must fix,
later Plan, accepted limitation, or not applicable. No real Provider, `.env`,
secret, user SQLite contents, or real network Tool may be used.

## Remaining Accepted Limitations

After this patch, Plan 2 still intentionally has synchronous non-streaming
execution, sequential Tools, no run list/poll/cancel/resume/retry, no strict
Provider request/response replay, no AgentStep/Trace or Agent-linked LLMCall
usage/cost, no Human Approval, mock-only Provider acceptance, deferred
`web_fetch`, and cooperative cancellation for blocking thread work. These are
documented bridge items or later-Plan capabilities, not incomplete work in this
patch.
